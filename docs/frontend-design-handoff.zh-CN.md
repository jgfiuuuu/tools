# Frontend Design Handoff

这份文档是给“只负责提出前端设计建议的其他 AI”准备的。  
目标不是让它改代码，而是让它在充分理解当前产品、代码结构、交互约束、用户偏好之后，给出不会跑偏的设计方案。

---

## 1. 项目定位

这是一个**科研文献工作台**，不是聊天机器人，也不是通用搜索页面。

核心任务是：

1. 用户输入一个研究问题。
2. 后端做多源学术检索、关键词拆分、候选召回与筛选。
3. 前端展示候选论文列表，让用户逐篇看链接和摘要，手动筛选。
4. 基于用户确认的论文，生成一份**研究报告 / 研究备忘录**。
5. 保存本地历史会话，支持后续回看、继续筛、删除。

一句话概括：  
**这是一个以“文献筛选 + 精读 + 研究判断”为中心的阅读型工作台。**

---

## 2. 当前前端技术现实

### 技术栈

- `Vue 3`
- `TypeScript`
- `Vite`
- 无 `router`
- 无 `Pinia/Vuex`
- 无组件库

### 入口文件

- `frontend/src/main.ts`
- `frontend/src/App.vue`
- `frontend/src/style.css`
- `frontend/src/services/api.ts`

### 构建与运行

- dev server: `vite`
- 默认端口：`5174`（见 `frontend/vite.config.ts`）
- build 输出目录：`frontend/build`
- API base URL 来源：`VITE_API_BASE_URL`，否则默认 `http://localhost:8000`（见 `frontend/src/services/api.ts`）

### 重要事实

当前前端几乎全部集中在单文件组件 `frontend/src/App.vue` 中：

- 模板
- 状态
- 业务交互
- 样式

都在一个文件里。

这意味着：

- 设计建议可以大胆，但**不要默认要求重写整套技术栈**。
- 如果建议涉及重构，优先是“拆组件、保留现有数据流”，而不是“换框架 / 换架构 / 重做成对话系统”。

## 3. 当前信息架构

当前 UI 是一个三栏工作台：

### 左栏：历史与入口

职责：

- 品牌标题
- 创建新调研会话
- 历史会话列表
- 流程日志

特点：

- 可收起
- 历史会话支持删除
- 视觉上应该是“轻索引”，不是主舞台

### 中栏：主舞台

职责：

- 当前会话标题与精简状态
- 错误 / 降级提示
- 检索子任务（默认折叠）
- 文献筛选条件
- 候选论文列表

特点：

- 这是整个产品的视觉中心
- 用户大部分时间应停留在这里
- 顶部固定区域已经做过一轮精简，但仍可继续优化

### 右栏：精读与报告

职责：

- 文献详情
- 研究报告

特点：

- 可收起
- 有 tab：`detail / report`
- 右栏是**次级工作区**
- 研究报告很重要，但仍然是**次级入口**，不应该压过中间的论文列表

## 4. 当前关键交互

### 4.1 会话流

1. 用户输入研究问题
2. 前端调用 `createSessionStream()`
3. 通过 SSE 接收检索与筛选过程事件
4. 会话完成后加载详细数据

相关代码：

- `frontend/src/App.vue`
  - `startResearch()`
  - `handleSessionEvent()`
  - `loadSessions()`
  - `loadSession()`

### 4.2 论文筛选流

用户可以对每篇论文：

- 查看详情
- 勾选纳入
- 改状态：`included / to_read / saved / excluded`

相关代码：

- `selectPaper()`
- `togglePaperSelection()`
- `setPaperStatus()`
- `patchPaper()`

### 4.3 报告流

用户基于已确认论文生成报告：

1. 点击“生成研究报告”
2. 前端调用 `generateReportStream()`
3. SSE 持续拼接 markdown
4. 生成完成后重新拉 session

相关代码：

- `generateReport()`
- `handleReportEvent()`
- `parseReportMarkdown()`

### 4.4 派生流

用户可以从当前论文继续派生新会话：

- 输入新方向
- 调用 `deriveSession()`

相关代码：

- `deriveFromActivePaper()`

### 4.5 历史记录管理

支持：

- 历史会话列表
- 选择历史会话
- 删除历史会话

相关代码：

- `openDeleteDialog()`
- `confirmDeleteSession()`

## 6. 当前 SSE 事件语义

会话创建与检索过程使用 SSE。

前端当前已经处理的事件包括：

- `session_created`
- `query_plan_generated`
- `query_generated`
- `source_skipped`
- `source_failed`
- `status`
- `papers_recalled`
- `subtask_completed`
- `paper_ranked`
- `screening_done`
- `report_chunk`
- `report_done`
- `error`

这些事件已经被映射到当前 UI 的以下区域：

- 左侧流程日志
- 中间检索子任务面板
- 中间论文工作台
- 右侧报告面板
- 错误 / 降级提示

设计建议必须默认这些“动态状态”仍然存在，不能只给静态页面。

## 7. 当前布局和滚动约束

### 桌面布局

默认是三列：

- 左：`288px`
- 中：`1fr`
- 右：`430px`

收起后左右栏会缩成：

- `74px`

相关 CSS 变量在 `frontend/src/App.vue`：

- `--left-width`
- `--right-width`
- `--bg`
- `--surface`
- `--surface-strong`
- `--line`
- `--line-soft`
- `--text`
- `--muted`
- `--accent`
- `--accent-strong`

### 响应式行为

- `> 1180px`：三栏
- `<= 1180px`：右栏变覆盖式抽屉
- `<= 960px`：整体堆叠

### 滚动行为

这是非常重要的现有约束：

- 中间论文列表单独滚动
- 右侧详情 / 报告面板单独滚动
- 选择论文后，右侧详情滚动会重置到顶部
- 报告目录导航只滚动右侧报告容器，不滚动整个页面

外部 AI 给设计方案时，**必须保留独立滚动容器思路**，否则会回到之前很难用的状态。

## 8. 当前视觉方向

当前前端已经从“普通 dashboard”往下面这个方向调整过：

### 目标气质

- 安静
- 高级
- 克制
- 阅读型
- 工具感强，但不要冷硬

### 参考来源

用户明确提出可参考：

- `https://yorha.in/`

但只是借鉴：

- 配色
- 材质
- 版面节制感
- 低饱和深色点缀

不是要做成：

- 强叙事 IP 页
- 游戏终端
- “档案馆 / 断案终端”
- 强世界观装置界面

### 当前已落地的视觉基调

- 暖米色 / 纸感背景
- 细线框
- 深棕黑色强调
- serif 标题
- 小号辅助文本
- 低优先级信息压缩为细粒度元信息

---

## 9. 用户已经明确表达过的偏好

下面这些不是“建议”，而是已经明确说过的方向。

### 必须遵守

1. **不要把它做成档案终端 / 断案终端。**
2. **不要做成吵闹的 SaaS 仪表盘。**
3. **中间区域是主舞台。**
4. **右侧首先是论文精读，不是大块元数据墙。**
5. **研究报告是重要但次级的入口。**
6. **低优先级信息必须缩小、合并、弱化。**
7. **历史会话不能无限增长，必须可删除。**
8. **BibTeX 比 Markdown 更重要。**
9. **左右收缩控件应该图形化，不要“打开/收起”大字按钮。**
10. **滚动要分离，中间列表与右侧阅读不能共用滚动。**

### 用户偏好的具体视觉倾向

- 喜欢 `NieR:Automata / YoRHa` 那种安静、克制、纸感、工业感
- 但不想要太强的世界观演出
- 更想要“简洁而高级”的学术工作台

---

## 10. 当前已解决的问题

为了避免外部 AI 提出已经解决过的东西，这里列一下现状：

1. 品牌已从旧名称迁移到 `文献妙妙屋`
2. 历史会话已有删除能力
3. 左右栏都已支持收起
4. 右侧详情与中间列表已做滚动分离
5. 右侧 report TOC 已限定在 report 容器内滚动
6. 主界面不再突出 Markdown 导出
7. BibTeX 保留在右侧细节区
8. 右侧元信息已从四个大块压成一行小 meta
9. 会话顶部信息已做过一轮压缩
10. 降级提示现在默认更紧凑

因此，新的设计建议应该继续往“更精炼、更耐看、更统一”推进，而不是把这些交互再推翻。

---

## 11. 目前仍值得优化的前端方向

这是希望外部 AI 重点发力的地方。

### A. 顶部固定区还能更干净

虽然已经压缩过，但中间顶部仍有进一步优化空间：

- 标题层级
- 副信息展示方式
- 操作按钮排布
- 固定高度控制

目标：  
**在不丢信息的前提下，再给中间论文列表多留一点可视高度。**

### B. 左侧入口区的密度还可以更优雅

尤其是：

- 新建会话表单
- 历史会话列表
- 流程记录

目标：  
**左栏更像轻量索引与启动区，不抢戏。**

### C. 右侧精读区可继续做成“阅读面板”

当前方向是对的，但还能继续优化：

- 标题、作者、meta 的节奏
- 摘要与 AI 判断的相对关系
- 引用区的呈现方式
- 派生新会话输入区的存在感

目标：  
**第一屏让用户自然进入阅读，而不是进入控制面板。**

### D. 报告面板仍然可以更次级、更优雅

它不能显得太像第二个主页面。

目标：  
**让报告更像“研究备忘录侧栏”，而不是主舞台竞争者。**

---

## 12. 设计提案时不要做的事

外部 AI 不要建议下面这些方向，除非明确说明只是可选分支：

1. 改成聊天式主界面
2. 加 landing page / hero marketing 页
3. 用大面积霓虹蓝紫科技风
4. 做成深色侦探终端 / 档案馆 / 指挥中心
5. 让右侧元信息重新变成大块卡片
6. 让研究报告取代中间论文列表成为主舞台
7. 增加复杂、强叙事、装饰性很重的世界观 UI
8. 默认引入大量新功能，超出当前产品范围
9. 要求彻底重构后端协议才能成立的设计
10. 把滚动重新绑到整页

## 13. 如果要给出设计方案，建议输出格式

希望外部 AI 给的是**可执行设计说明**，而不是泛泛点评。

建议它输出：

### 1. 设计目标

用 3 到 6 句话定义新方案的气质与核心原则。

### 2. 桌面布局

明确说明：

- 左栏怎么处理
- 中栏顶部怎么压缩
- 中栏列表怎么留白
- 右栏详情 / 报告怎么组织

### 3. 移动端 / 窄屏策略

至少给出：

- 侧栏收纳方式
- 主列表优先级
- 右栏如何抽屉化

### 4. 组件级别建议

最好逐项点评：

- 左栏品牌区
- 新建会话区
- 历史会话项
- 中栏顶部
- query board
- filter bar
- paper row
- detail hero
- abstract block
- cite block
- report block

### 5. 视觉 token 建议

例如：

- 颜色
- 字号层级
- 间距系统
- 边框粗细
- 阴影策略
- 圆角策略

### 6. 优先级说明

要区分：

- 必改
- 推荐改
- 可选优化

这样方便后续按阶段执行。

## 14. 推荐交给外部 AI 的任务边界

建议你让外部 AI 做的是：

- 风格拆解
- 版式优化
- 信息层级优化
- 组件视觉策略
- 移动端适配建议

不建议让它做的是：

- 编造新后端功能
- 改 API 协议
- 重写整个产品逻辑
- 把产品重新定义成聊天机器人

## 16. 给执行实现的一句提醒

后续如果外部 AI 给出了设计方案，执行时应优先保证：

1. 不破坏当前 API 契约
2. 不破坏独立滚动
3. 不破坏会话 / 论文 / 报告的数据流
4. 不把产品带偏成聊天 UI 或终端 UI

---

## 17. 建议同时提供给外部 AI 的前端材料

结论：**要，最好同时给它读关键前端文件。**  
如果只给这份 handoff 文档，外部 AI 能做出方向判断，但很容易停留在抽象层；如果你希望它给出能被 GPT 直接执行的方案，至少还应让它同时阅读当前前端结构与样式入口。

### 最少应同时提供

- `frontend/src/App.vue`
- `frontend/src/style.css`
- `frontend/src/services/api.ts`
- `frontend/src/main.ts`
- `frontend/vite.config.ts`

### 如果你能再多给一点，上限建议是

- 当前页面截图
- 桌面端截图一张
- 窄屏 / 移动端截图一张
- 你最不满意的区域截图一张

### 为什么这些文件要给

- `App.vue`：让它理解当前三栏布局、组件边界、事件流、哪些地方耦合在一起
- `style.css`：让它判断当前 token、基础排版、重置样式、全局变量和视觉基线
- `api.ts`：让它知道哪些数据结构和交互必须保留，不至于提出脱离现实的组件设计
- `main.ts`：确认应用入口和全局挂载方式
- `vite.config.ts`：帮助它理解构建与运行环境，避免提出依赖框架级改造的建议

### 推荐你对外部 AI 的要求

如果它能读代码，就要求它先按这个顺序看：

1. 先读本 handoff 文档
2. 再读 `frontend/src/App.vue`
3. 再读 `frontend/src/style.css`
4. 再读 `frontend/src/services/api.ts`
5. 最后结合截图给出设计方案

### 不建议一股脑全给的内容

- 不要把整个 `frontend/node_modules` 或 build 产物丢给它
- 不要让它从全仓库随意发散
- 不要把任务变成“读完整个代码库再做设计”

目标不是让它做 repo archaeology，而是让它**足够理解当前前端约束后，输出一份高质量、可执行、不会跑偏的 UI 设计方案**。
